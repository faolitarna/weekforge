# Architectural Decision Log

This log records technical decisions that resolve ambiguity, deviate from the initial spec, or introduce new constraints.

| Date | ID | Decision | Context | Spec Impact |
|---|---|---|---|---|
| 2026-04-10 | DEC-001 | Decompose Monolithic Spec | The `Migration-Spec.md` was too large (399 lines) to review and track effectively, leading to SDD failures. | Triggered the creation of the multi-file spec suite and this decision log. |
| 2026-04-11 | DEC-002 | Consolidate Phase 0 contracts into WF-00 | Phase 0 infrastructure contracts (Notion Tool Layer, Model Config, CLI, Secrets, Tooling Standards) were scattered across conceptual descriptions with no implementation-level detail. Cross-cutting stubs (WF-OBS, WF-TEST, WF-PROMPT) were incorrectly tagged as Phase 0. | WF-00 bumped to v1.1 with six new sections (§3-§8). Cross-cutting stubs re-phased to post-Phase 0. | 2026-04-14 | DEC-003 | Extract data_source_id for Notion DB Queries | Notion SDK 3.0.0 (API v2025-09-03) dropped `databases.query` and formally decoupled Databases (views) from Data Sources (records). | The `query(database_id)` contract in step-0b remains stable, but the `notion.py` tool implementation internally refactored to fetch the database, extract `data_source_id`, and query it via `data_sources`. |
| 2026-04-15 | DEC-004 | Migrate from LangGraph to Pydantic AI | Weekforge is a single-agent sequential workflow system (Agentic Level 2). LangGraph's graph orchestration, multi-agent coordination, and streaming infrastructure add dependency weight (langgraph, langgraph-checkpoint-sqlite, langchain-core + transitive deps) without proportional value. Every workflow is a sequential pipeline with at most one loop — plain Python control flow suffices. Pydantic AI provides LLM integration with structured output validation, dependency injection, and model-agnostic bindings in a lighter package. | All step specs (0a–4), all reference docs (architecture, patterns, state-schema). pyproject.toml dependency swap. Directory rename `graph/` → `workflows/`. New modules: `checkpoint.py`, `hitl.py`, `agents/`. Step 0a status reset to 🔄 (re-implementation needed for new foundation). |
| 2026-04-17 | DEC-005 | Caveman-lite style directive as global env toggle | Coach workflow (phase 1+) returns user-facing prose; output tokens are the dominant cost on `gpt-5.4`. A style directive that strips filler + hedging while keeping polish reduces cost without degrading output. Scoped as a single global env flag (`CAVEMAN_MODE`), hardcoded directive text (no external file / YAML), lite register only (full/ultra would break user-facing polish). Applied at system-prompt composition time so structured-output schemas remain byte-identical. | New reference doc [reference/prompt-style.md](./reference/prompt-style.md). step-0c Part 2 gains a composer-routing acceptance bullet. Future phase-1+ agent definitions route their `system_prompt` through the composer. No change to `LLMProfile` or `run_with_metadata`. |
| 2026-04-19 | DEC-006 | Step-1 restructure: split into 1a–1d, drop `run_log`, single HITL accept gate, Tier-0 checkbox analysis, Pydantic AI `instructions=` pattern | Original [step-1-extraction.md](./steps/step-1-extraction.md) was a thin monolithic spec that did not recreate `source-material/.claude/commands/summarize_week.md` (missing persona/guardrails/user-profile wiring, missing DB IDs, wrong HITL shape, deterministic checkbox analysis treated as LLM work, no summary-acceptance gate). Review spawned multiple design decisions bundled into this entry: (a) split step-1 into sub-steps 1a–1d so each is independently implementable by a weaker model; (b) `run_log` DB retired (legacy workflow-state storage superseded by SQLite checkpoint); (c) planned-week persistence moved to a `Plan` property on the `training_week_summaries` row (step-2 owns the write) — enables delta analysis; (d) HITL collapsed from two fetch-verify gates to one accept-summary gate with highlights+trend and feedback loop (reuses [e2e.py](../../src/weekforge/workflows/e2e.py) pattern); (e) checkbox + delta analysis moved to Tier-0 `tools/extraction.py`; (f) agent prompts composed via Pydantic AI `instructions=` (static persona+guardrails, cacheable prefix) + `@agent.instructions` decorators (dynamic user-profile and Tier-0 facts via `RunContext`) — NOT `system_prompt=`, per Pydantic AI docs recommendation for feedback-loop workflows; (g) persona+guardrails as local markdown in `src/weekforge/prompts/`, user-profile as single-row Notion DB with typed properties + prose body. | [step-1-extraction.md](./steps/step-1-extraction.md) rewritten as index; new specs 1a–1d. [_index.md](./_index.md) dashboard rows added. [step-2-planning.md](./steps/step-2-planning.md) needs a follow-up update: at plan approval, pre-create the `training_week_summaries` row with `Plan=<markdown>`, `Summary=<empty>`. [e2e_agent.py](../../src/weekforge/agents/e2e_agent.py) flagged for small cleanup (migrate `system_prompt=` → `instructions=`) — not blocking for step 1. New files to be created under `src/weekforge/prompts/`, `src/weekforge/config/user_profile_loader.py`, `src/weekforge/models/user_profile.py`, `src/weekforge/tools/week_prefix.py`. `.env.template` + `env.py` gain four training DB ID vars; `NOTION_DB_RUN_LOG` NOT added. |
| 2026-04-19 | DEC-007 | User profile: Notion DB → single Notion page | The DB approach required 6 typed properties with exact names (`Name`, `Training Age (years)`, `LTHR`, `Max HR`, `Conditions`, `Active`), an Active checkbox filter, and 4 property-parsing helpers — all for a single row whose typed fields were never consumed structurally. The only downstream consumer (`_inject_user_profile` in step-1c) injects `prose_markdown` into the LLM. Typed fields (`hr_anchors`, `training_age_years`, `conditions`) appeared only in a cosmetic CLI panel. Simplifying to a single page eliminates ~80 lines of loader/model code, removes the DB schema setup burden, and makes the profile trivially extensible (user just edits their page). Supersedes DEC-006 item (g). | step-1a: user profile section rewritten (page setup replaces DB schema); `UserProfile` model simplified to `{page_id, markdown}`; `HRAnchors` deleted; `NOTION_DB_USER_PROFILE` → `NOTION_USER_PROFILE_PAGE_ID` in env. step-1c: `prose_markdown` → `markdown`. step-2: `PlanningDeps.user_profile` type changed from `dict` to `UserProfile`. |
| 2026-04-19 | DEC-008 | Notion API integration hardening (post step-1d) | First end-to-end run of `weekforge summarize-week` against a real workspace surfaced five integration bugs: (a) `data_sources.query` rejects `filter_properties` (schema error); (b) Notion stores the `Week` property as a plain numeric string (`"1"`), not a zero-padded prefix (`"W01"`); (c) the title property is not guaranteed to be named `"Title"` and is exposed in either the public-API schema shape or the internal data-source schema shape; (d) `blocks.children.append` is capped at 100 children per request, which the renderer exceeded on busy weeks; (e) no cost/observability surface was emitted at workflow completion. | `tools/notion_api_gateway.py` — drop `filter_properties` in query; add dynamic title-property discovery covering both schema shapes; client-side filter by rich_text `plain_text`. `workflows/extraction.py` — week filter parses `int(week_prefix[1:])`, batch block appends in chunks of 100, print `RunCost` summary panel on `done`, log created PLAN_STATE page id. `tools/raw_session_collector.py` — tolerate malformed block payloads via `.get()`. No spec contract change; step-1a/1d narratives updated with batching + discovery details. |
