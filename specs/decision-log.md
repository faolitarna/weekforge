# Architectural Decision Log

This log records technical decisions that resolve ambiguity, deviate from the initial spec, or introduce new constraints.

| Date | ID | Decision | Context | Spec Impact |
|---|---|---|---|---|
| 2026-04-10 | DEC-001 | Decompose Monolithic Spec | `Migration-Spec.md` too large (399 lines) to review and track effectively; caused SDD failures. | Multi-file spec suite created; this decision log added. |
| 2026-04-11 | DEC-002 | Consolidate Phase 0 contracts into WF-00 | Phase 0 infrastructure contracts scattered across conceptual descriptions with no implementation detail. Cross-cutting stubs (WF-OBS, WF-TEST, WF-PROMPT) incorrectly tagged as Phase 0. | WF-00 bumped to v1.1; cross-cutting stubs re-phased to post-Phase 0. |
| 2026-04-14 | DEC-003 | Extract data_source_id for Notion DB queries | Notion SDK 3.0.0 dropped `databases.query`; decoupled Databases (views) from Data Sources (records). | `notion.py` internally fetches database, extracts `data_source_id`, queries via `data_sources`. Public contract unchanged. |
| 2026-04-15 | DEC-004 | Migrate from LangGraph to Pydantic AI | Weekforge is a single-agent sequential workflow (Agentic Level 2). LangGraph's graph orchestration adds dependency weight without proportional value. Pydantic AI provides LLM integration with structured output, dependency injection, and model-agnostic bindings in a lighter package. | All step specs and reference docs updated. `graph/` → `workflows/`. `pyproject.toml` dependency swap. |
| 2026-04-19 | DEC-005 | Step-1 split into sub-steps 1a–1d | Monolithic step-1 spec bundled extraction, context loading, summarization, and Notion write — too large for a single implementation unit. | `step-1-extraction.md` rewritten as index; sub-specs 1a–1d created. |
| 2026-04-19 | DEC-006 | `run_log` DB retired; Plan persisted on `training_week_summaries` row | `run_log` was legacy workflow-state storage superseded by SQLite checkpoint. Moving the Plan field to `training_week_summaries` collocates planned vs. actual data and enables delta analysis in step-2. | `NOTION_DB_RUN_LOG` dropped from env. step-2 owns Plan write at approval time. |
| 2026-04-19 | DEC-007 | HITL collapsed to single accept-summary gate | Two fetch-verify gates added friction without value. Single accept+feedback loop on the final summary matches the real user decision point. | step-1d owns HITL gate; earlier sub-steps produce outputs only. |
| 2026-04-19 | DEC-008 | Pydantic AI `instructions=` over `system_prompt=` for agents | Pydantic AI recommends `instructions=` for feedback-loop workflows: static prefix is cacheable; dynamic context injected via `RunContext` at call time. `system_prompt=` bypasses this split. | All agent definitions use `instructions=` + `@agent.instructions` decorators. |
| 2026-04-19 | DEC-009 | User profile: Notion DB → single Notion page | Typed DB properties were never consumed structurally — only downstream use injected prose into LLM. Single page eliminates schema setup burden and ~80 lines of loader code. Supersedes DEC-008 item (g). | `UserProfile` model simplified to `{page_id, markdown}`; `NOTION_DB_USER_PROFILE` → `NOTION_USER_PROFILE_PAGE_ID` in env. |
| 2026-04-25 | DEC-010 | Agent renames, prompt extraction, model class split | Post step-1 completion refinements. (a) Agent names aligned to workflow verbs: `summarize_agent` → `summarize_week_agent`, `plan_state_agent` → `update_plan_state_agent`. (b) All embedded LLM text extracted to `prompts/*.md`; `prompt_composer` parameterized with `task: Prompt`. (c) `summarize_week_agent` switched to `fast` profile (gpt-5.4-mini, no reasoning overhead) — extraction task doesn't need reasoning. `update_plan_state_agent` stays on `reasoning` (trend interpretation). (d) LLM profiles expanded: gpt-5.4-mini, gpt-5.4-medium, gpt-5.4-low. (e) `update-plan-state-task.md` strengthened with field-by-field specs. (f) WeekSummary filtered before plan_state agent to strip noise. | Specs 1a–1f use old agent names in design sections (historical); Implementation Status section on step-1-extraction.md documents current names. `architecture.md` LLM profiles table updated. |
